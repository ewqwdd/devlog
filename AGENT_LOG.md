# Використання агентів

Основний coding agent — **Claude Code** (CLI). Робоча модель — **Opus 4.8**, на старті
пробував **Fable 5** (деталі нижче). Для редизайну UI окремо використовував **Claude Design**.

## Підготовка проєкту

Вирішив використовувати патерни SDD (spec-driven development) для роботи з агентом, щоб
підійти структурно і не перетворити проєкт на хаос.

Спочатку сформував DESIGN.md: голосом повністю описав функціонал і технології, які буду
використовувати, в чаті з Claude побрейнштормив, покращив документацію і попросив його
сформувати фінальний варіант.

На основі DESIGN.md з Claude сформував ROADMAP.md, зробив пару мінімальних ручних правок.

DESIGN.md і ROADMAP.md — це source of truth для проєкту, ядро SDD.

## Налаштування середовища

Сформував початковий CLAUDE.md з патернами розробки та правилами написання коду.
Це важливо, бо в розробці багато агентних сесій, і всі вони мають дотримуватись одних
правил — такий код набагато простіше масштабувати і робити йому рев'ю.
Там описані правила по архітектурі проєкту, декомпозиції та best practices.
Opus часто створює зайві компоненти, не дотримується архітектури проєкту та ігнорує
типізацію, тому йому треба чітко сказати цього не робити.

Після цього CLAUDE.md оптимізовувався декілька разів, з'являлись нові правила.
Фінальний варіант сильно скорочений і більш конкретний, щоб Claude було простіше його
дотримуватись.

### Вибір моделі

Нещодавно вийшла нова модель Fable 5, якраз хотів її протестити на цьому проєкті, тому
спочатку використовував її. Через день зрозумів, що це overkill: можливо, це мало б сенс,
якби я робив проєкт повністю за один промпт, але я рухався поетапно
(1 сесія агента = 1 фаза роадмапу). В такому режимі різниці в якості між Fable та Opus
не було, при цьому Fable витрачав у 2 рази більше токенів і в 1.5 рази більше часу.
Повернувся на Opus 4.8.

### Skills

- **playwright-cli** — додав, бо Claude в процесі роботи іноді використовував
  playwright-mcp, глобально встановлений у мене, а використання через skill + CLI
  витрачає менше токенів.
- **shadcn** — готові скіли від UI-бібліотеки, яку використовував; ніколи до цього їх не
  пробував — хотів спробувати. Поки ніякого імпакту від них не побачив: і без них Claude
  нормально працював з бібліотекою і знаходив потрібні компоненти.
- **context7 (plugin)** — надає доступ до актуальної документації популярних бібліотек.
  Важлива штука, бо технології оновлюються, а модель може бути навчена на застарілих
  даних.
- **superpowers** — найпопулярніший фреймворк для SDD. Повністю скопіював його в
  `.claude` і зробив декілька модифікацій:
  - у spec додав обов'язковий verification plan з e2e-тестами, щоб агенти були більш
    автономними і перевіряли задачі реальними тестами;
  - для subagent-driven-development розбив задачі на 2 типи: механічні та логічні.
    Механічні пропускають фазу рев'ю — це максимально прості задачі, в яких майже немає
    мислення. Зроблено тому, що на кожну задачу після виконання запускаються 2 рев'юери,
    через що виконання затягувалось на 40+ хвилин, хоча задача могла бути невеликою;
  - додав cap на максимум 8 задач у плані, бо роадмап і так добре розбитий;
  - зробив так, щоб після виконання він відмічав етап роадмапу як виконаний.

Модифікація з обов'язковими e2e-тестами дала більше надійності й автономності агентам:
з мого досвіду більшість помилок агентів — саме на frontend, бо не все можна покрити
unit-тестами.
Всі інші модифікації пов'язані з тим, що задачі виконувались дуже довго. Самі агенти
працювали швидко — bottleneck був у логіці роботи superpowers: він дуже надійний, але
іноді за це доводиться платити часом виконання.
Перші фази виконувались 50–60 хв, після правок — 30–40.

### Hooks

Тільки хук від superpowers для ініціалізації та скрипт, який тегає мене в Telegram,
якщо я пропустив момент, коли виконання зупинилось чи закінчилось.

## Флоу роботи

- Генерація спеки через `/brainstorming` за моїм описом + ROADMAP.md + DESIGN.md.
- Human review спеки — по суті найважливіша частина: спека — це фактичний контракт з
  агентом, він чітко зробить те, що там написано, тому більшість потенційних помилок
  виправлялась саме на цьому етапі.
- `/writing-plan` + виконання за планом — повністю автономна фонова робота на 30+ хв.
- Під час виконання я генерував спеки для наступних задач або робив code review та
  рефактор попередніх. Агент-виконавець працював в окремому git worktree, тому я міг
  спокійно робити будь-які правки в main і максимально ефективно використовувати свій
  час.

## Що генерував агент і де довелось втручатись

Весь код, включно з UI (компоненти, лейаут, стилізація на Tailwind + shadcn/ui),
згенерований агентом. Ручних правок було зроблено дуже мало: завдяки детальній спеці й
дизайну логічних помилок майже не було.

Основні помилки, які я виправляв:

- **Архітектурні** — файли в неправильній папці. Після виправлення оновлював CLAUDE.md,
  щоб помилка не повторювалась. Наприклад:
  - агент створив компонент всередині сторінки в папці `_components`, хоча потім
    використав його на іншій сторінці;
  - агент створив React Query хуки inline всередині компонента — я вважаю, що логіку
    хуків зручніше виносити окремо.
- **UI** — не сподобався дизайн від агента, тому згенерував редизайн у Claude Design і
  запустив паралельно окремого агента, який зробив повний редизайн за новим макетом.

Явні логічні помилки, які знайшов і виправив:

- для зміни порядку задач робився окремий запит у БД на кожну задачу — замінив на один
  запит для всіх задач;
- drag-and-drop дивно стрибав — це загалом очікувано, бо агенту навіть з Playwright
  неможливо протестити анімації.
